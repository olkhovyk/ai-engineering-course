Fine-tuning vs RAG

Як насправді давати моделі нову інформацію

💡 Частий неправильний розуміння: Fine-tuning = давання моделі нової інформації

На жаль, це частий спотик 😊 Fine-tuning — це не про додавання знань, а про зміну стилю, поведінки та закономірностей у відповідях. Fine-tuned модель використовує ті ж самі ваги і не має доступу до ваших специфічних даних під час роботи.

Якщо вам потрібно дати моделі доступ до нових даних (документи, бази даних, специфічна інформація), то RAG — це ваш вибір! 📚

🧪 Порівняння моделей: Ollama vs OpenAI

Одне й те саме запитання, різні моделі. Оцінка через LLM-as-Judge (GPT-4)

📝 Запит для порівняння:

🦙 Ollama (Llama 2 7B)

Обидва підходи можуть працювати. Fine-tuning вимагає great training data, а RAG вимагає good retrieval.

  • Fine-tuning: Тренування embedding моделі на domain data. Швидко на інференції.
  • RAG: Пошук документів + LLM. Гнучче оновлювати.
  • Затримка: Fine-tuning швидше, RAG повільніше через retrieval step.
  • Масштабування: Обидва можуть масштабуватись до 10M користувачів.

Вибір залежить від ваших вимог. Fine-tuning потребує GPU для тренування. RAG потребує доброго search index. Розважайте про trade-offs і виберіть те, що краще для вас.

🤖 OpenAI (GPT-4o-mini)

Fine-tuned embeddings є кращим вибором для ваших вимог, але RAG з hybrid подходом дає найкращий balance:

  • Fine-tuned embeddings (top-1 recall@10 ≤ 30%): Тренуйте on domain pairs (user_id, item_id, click/purchase signal). Це потребує 100K+ пар. Кожна інференція: ~5мс (embedding lookup). Але обновлення коли додаються нові товари потребує переindex всього (дорого).
  • RAG Hybrid (Fine-tuned dense + BM25 sparse retrieval): 1) Dense retrieval з fine-tuned BERT (top-1000), 2) BM25 на метаданих (категорія, бренд). Merge результатів (50% dense + 50% sparse). LLM ranking на top-100. Latency: ~20-40мс. Обновлення товарів: тільки додай в index (~1мс).
  • Infrastructure: Вектор DB (Pinecone, Weaviate, Qdrant). Для 10M items × 768dim embeddings = ~30GB VRAM на GPU. Redis для caching top-1000 retrieval результатів (99% cache hit).
  • Cost: Fine-tuning GPU training: $200-500. Serving: Qdrant self-hosted ~$2K/mo або Pinecone ~$500-5K/mo залежно від scale. RAG LLM ranking: +0.5-1cent per request.
  • Quality vs latency trade-off: Fine-tuning only: 30% recall але 5мс. RAG hybrid: 45-50% recall але 25мс. Для e-commerce 25мс це OK, user perception <500мс.

Recommendation: Почніть з fine-tuned embeddings (baseline quick wins), потім додайте BM25 retrieval для гібридного підходу. Це дає вам гнучкість RAG з швидкістю fine-tuning. А/B тест на 10% користувачів для вимірювання conversion lift.

🎯 Висновок LLM-as-Judge:

Ollama (7B): Дає загальну ідею "fine-tune або RAG", але не розуміє scale реалізації (latency constraints, cost, infrastructure choices). Відповідь поверхнева.

OpenAI (GPT-4o-mini): Надає production-ready рішення: конкретні вибір (Pinecone, Redis, Qdrant), cost estimates ($200-5K/mo), trade-off аналіз (recall@10 vs latency), гібридний підхід.

Різниця у якості: Ollama не розуміє "10M users" як системний constraint. OpenAI проектує систему яка масштабується. Для production → OpenAI необхідний. Для quick sketch → Ollama OK.

🎯 LoRA Fine-tuning: більш детально

✅ LoRA на Open Source моделях

Приклади: Llama 2, Mistral, Falcon

Як працює:
  • Замерзаємо всі ваги моделі
  • Додаємо маленькі матриці (A × B) до певних шарів
  • Тренуємо тільки ці матриці (~0.1-1% параметрів)
  • Зберігаємо: оригінальна модель + LoRA ваги
Переваги:
  • ✓ 10-20GB RAM достатньо
  • ✓ Тренування на RTX 4090 (~2-4 години)
  • ✓ LoRA адаптер = 10-100MB (легко ділитися)
  • ✓ Можна мати багато LoRA адаптерів для однієї моделі
Приклад з peft бібліотекою: 
from peft import LoraConfig, get_peft_model

config = LoraConfig(
    r=16,  # LoRA rank
    lora_alpha=32,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.05
)

model = get_peft_model(model, config)
# Тренуєм тільки на ПК! 🎉

  • Можна fine-tune на своїх даних
  • Модель депортується на OpenAI серверах

OpenAI Fine-tuning API приклад: 
import openai

# Завантажимо дані
with open("training.jsonl") as f:
    file_id = openai.File.create(
        file=f,
        purpose="fine-tune"
    ).id

# Fine-tune (дорого! 💰)
job = openai.FineTune.create(
    training_file=file_id,
    model="gpt-3.5-turbo",
    n_epochs=3
)

🎓 Fine-tuning

Що це?

Переподготовка частини або всіх параметрів моделі на ваших даних.

1
Ви передаєте приклади (prompt + відповідь)
2
Модель коригує свої ваги на основі ваших даних
3
Отримуєте нову модель зі змінореми ваги
4
Розгортаєте цю нову модель на сервері

Що змінюється?

✅ Стиль письма та тон відповідей

✅ Закономірності та формати (структура JSON, таблиці)

✅ Припинення та біази моделі

❌ Де модель дістає інформацію (ваги моделі)

✓ Переваги

  • Швидкі відповіді (все в вагах)
  • Краща якість при правильному наборі даних
  • Модель "навчається" вашому стилю
  • Можна fine-tune на недорогих моделях

✗ Недоліки

  • Потребує багато часу та даних (сотні-тисячі прикладів)
  • Дорого комп'ютерно (GPU для тренування)
  • Складно оновити інформацію (потрібна перетренування)
  • Лішити можна "забути" старі знання

Коли використовувати:

  • Потрібна специфічна поведінка/стиль
  • Є достатньо даних для тренування
  • Затримка відповіді критична
  • Часто потрібні відповіді однакового формату

📚 RAG (Retrieval-Augmented Generation)

Що це?

Отримання релевантних документів з бази даних і передача їх в prompt до моделі.

1
Користувач ставить запитання
2
Система шукає релевантні документи в базі
3
Вставляє документи в системний prompt
4
Модель дає відповідь на основі документів

Що змінюється?

✅ Кто модель має доступ до інформації

✅ Яку контекст отримує модель

✅ Можна оновити базу без перетренування

❌ Самі ваги моделі залишаються сталими

✓ Переваги

  • Легко оновити інформацію (просто додайте документ)
  • Модель не забуває старі знання
  • Недорого розпочати (не потрібно GPU)
  • Масштабується до великих баз даних
  • Відповіді мають джерела (transparency)

✗ Недоліки

  • Повільніше (потрібно шукати документи)
  • Залежить від якості пошуку (может не знайти найкращий документ)
  • Більший prompt = більше токенів = більше коштів
  • Модель повинна "слідувати" контексту

Коли використовувати:

  • Часто оновлюється інформація
  • Велика база даних знань
  • Потрібні джерела відповідей
  • Хочете уникнути перетренування

Деталізоване порівняння

Час виконання
Інференція: Швидка (мс) — все в моделі

Інференція: Повільніша (сек) — пошук + генерація
Підготовка: Швидка (індекс документів)
Оновлення даних
Потрібна перетренування моделі. Можна зробити incremental fine-tuning, але це складно.
Просто додайте новий документ до бази. Миттєво доступно.
Кількість прикладів
Сотні-тисячі приклад для хорошої якості. Чим більше, тим краще.
Почніть з будь-якої кількості документів. Масштабується легко.
Вартість
Велика ($): GPU для тренування, потім розгортання своєї моделі
Низька ($): Платите за API запити + пошук документів

Ні. Модель не забуває. Контекст завжди свіжий.
Transparency
Чорна скринька. Невідомо, звідки модель дістала інформацію.
Цілком прозора. Видно, з яких документів модель взяла інформацію.
