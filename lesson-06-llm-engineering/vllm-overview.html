❓ Що таке vLLM?

vLLM — це швидка та легка бібліотека для LLM inference та serving. Розроблена в Sky Computing Lab UC Berkeley, вона виросла в один з найактивніших open-source AI проектів із тисячами контрибюторів від академічних інститутів та компаній.

🎯 Основна ідея: PagedAttention

vLLM використовує PagedAttention — революційну техніку ефективного керування пам'яттю для attention key-value. Замість того щоб виділяти пам'ять блоками, vLLM використовує сторінки (як в операційній системі), що дозволяє:

  • 💾 Зменшити фрагментацію пам'яті — більше місця для вибірки
  • 🚀 Continuous batching — обробляти запити разом без очікування
  • ⚡ 23x вищу пропускну здатність порівняно з традиційними методами
  • 📉 Нижча latency — faster responses

🏗️ Архітектура vLLM

vLLM побудована з 6 основних шарів, кожен з яких має специфічну роль у системі:

📊 User API Layer

Роль: Вхідна точка для клієнтів

Функції:

  • OpenAI-compatible API — /v1/completions, /v1/chat/completions для прямої сумісності з OpenAI SDK
  • gRPC — high-performance RPC для внутрішніх сервісів
  • Anthropic Messages API — альтернативний API формат
  • Парсинг запитів, валідація параметрів, error handling

Приклад: POST /v1/completions { model: "mistral-7b", prompt: "...", max_tokens: 100 }

🎯 Request Scheduler

Роль: Оркестрація потоку запитів

Функції:

  • Continuous Batching — автоматично комбінує новоприйшлі запити без очікування завершення попередніх
  • Request Prioritization — пріоритизація коротких запитів або VIP клієнтів
  • Load Balancing — розподіл навантаження між GPU
  • Adaptive Batching — динамічний розмір батчу залежно від довжини запитів

Приклад: Замість чекати 5 запитів на завершення, начебто вже починаємо обробляти 5-й запит разом з обробкою 1-4

⚡ PagedAttention Engine

Роль: Революційне управління пам'яттю для attention

Функції:

  • Paged Memory Management — замість монолітних блоків використовує «сторінки» (як OS), це зменшує фрагментацію пам'яті
  • KV Cache Optimization — ефективне кешування Key-Value пар для уникнення переобчислення
  • Token-wise KV Cache — зберігає KV для кожного токена окремо сторінками
  • Memory Sharing — коли батч містить схожі префікси (fork points), KV буферується один раз

Приклад: замість 100MB для одного запиту, вмещаємо 10 запитів на 150MB разом завдяки паджингу та дедублікації

🔌 Kernel Executors

Роль: Оптимізовані обчислення для LLM операцій

Функції:

  • FlashAttention — efficient attention algorithm з меньшою I/O
  • FlashInfer — розширена версія для inference з різними seq lengths
  • CUTLASS — оптимізовані GEMM (matrix multiply) kernels
  • torch.compile — AOT компіляція графів операцій
  • TRTLLM-GEN — TensorRT-LLM kernels для максимальної швидкості
  • MoE Kernels — спеціалізовані для Mixture-of-Experts архітектур

Приклад: FlashAttention замість стандартного compute скорочує O(N²) пам'ять на O(N)

💾 Memory Manager

Роль: Управління виділенням та звільненням GPU пам'яті

Функції:

  • Dynamic Allocation — виділяє пам'ять за потребою під час inference
  • Block Manager — управління фіксованими блоками (сторінками) пам'яті
  • KV Cache Management — track яка пам'ять використовується для якого батчу
  • Garbage Collection — звільнення пам'яті завершених запитів
  • Memory Reuse — переиспользування звільненої пам'яті без перевиділення
  • GPU Memory Utilization — коефіцієнт (0.0-1.0) для контролю скільки VRAM використовувати

Приклад: GPU з 24GB VRAM при 0.9 утиліації використає ~21.6GB для KV cache, моделі ваг, та активацій

🖥️ Hardware Backend

Роль: Абстракція апаратного забезпечення

Функції:

  • NVIDIA CUDA — оптимізовані kernels для NVIDIA GPUs (A100, H100, RTX)
  • AMD HIP — support для AMD MI300, MI250 та інших
  • CPU Kernels — fallback для x86, ARM, PowerPC CPUs
  • TPU Support — Google TPU v4, v5 (експериментально)
  • Intel Gaudi — Intel специалізовані AI chips
  • Plugin System — власні backends для нестандартного ПЗ

Приклад: Один і той же vLLM код запускається на NVIDIA A100, AMD MI300, чи Apple Silicon без змін

🔄 Потік даних через архітектуру

1. User sends request → API Layer парсить JSON

2. Scheduler обирає батч → вирішує які запити обробляти разом

3. Memory Manager виділяє → KV cache для цього батчу

4. PagedAttention Engine → обчислює attention за допомогою сторінок

5. Kernel Executors → запускають оптимізовані операції (FlashAttention, GEMM)

6. Hardware Backend → виконує операції на GPU/CPU/TPU

7. Response sent back → токени, embedding, або інший output

☁️ Де насправді деплоять vLLM? (AI Engineer Level)

Коли ви виходите за межі свого ноутбука, vLLM стає основним вибором для трьох популярних сценаріїв:

🚀 A. Хмарні GPU-хостинги

Платформи: RunPod, Lambda Labs, Vast.ai, Crusoe

Ідея: Ви орендуєте одну або кілька GPU карток на годину

Як працює: Запускаєте Docker-контейнер з vLLM → він видає вам URL → працює як OpenAI API

Приклад:

docker pull vllm/vllm-openai:latest
docker run --gpus all -p 8000:8000 \
  vllm/vllm-openai:latest \
  --model mistralai/Mistral-7B-v0.1

Ціна: $0.40–$4+ за годину (залежно від GPU)

RunPod A100: ~$0.67/год | H100: ~$2.49/год

Найкраще для: Стартапи, MVP, тестування в production-like умовах

Плюси: ✓ Гнучкість, ✓ Платиш скільки користуєшся, ✓ Швидкий старт

Мінуси: ✗ Холодний старт (30-60 сек), ✗ Дорого за 24/7, ✗ Залежиш від гостинга

⚡ B. Serverless GPU (True Pay-Per-Use)

Платформи: Modal, Replicate, Beam, Baseten

Ідея: Ви не платите за оренду сервера 24/7, а тільки за **секунди, коли код реально працює**

Як працює: Ваш агент прокидається, запускає inference, платить лише за те час, потім "засинає"

Приклад (Modal):

@app.function(gpu="A100")
def summarize_meeting(audio_file):
  # vLLM стартує, робить inference, ланчує
  response = vllm_client.complete(...)
  return response

Ціна: Зазвичай дешевше RunPod через паралелізм

Найкраще для: Автоматизація (боти, агенти, які працюють не 24/7), eventos-driven системи

Плюси: ✓ Платиш тільки за робочий час, ✓ Auto-scaling out-of-box, ✓ Жодного DevOps

Мінуси: ✗ Cold start все ще 30-60 сек, ✗ Дорого за high-throughput, ✗ Vendor lock-in

🖥️ C. Bare Metal сервер (Приватність + Контроль)

Платформи: Власний датацентр, private cloud, on-premise

Ідея: Велика компанія (банк, медицина) купує сервер з GPU і запускає vLLM безпосередньо

Конфігурація: Ubuntu Linux + NVIDIA/AMD драйвери + vLLM як бекенд для всієї компанії

Приклад (фінтех):

# Bare metal: 8x A100 на Ubuntu 22.04
vllm serve meta-llama/Llama-2-70b-hf \
  --tensor-parallel-size 8 \
  --max-model-len 4096 \
  --port 8000

Ціна: CapEx ~$200K (купівля) + OpEx ~$5K/місяць (електрика, охолодження)

Найкраще для: Банки, медицина, державні установи, великі корпорації

Плюси: ✓ Приватність 100%, ✓ No vendor lock-in, ✓ Низька latency (локально), ✓ Контроль

Мінуси: ✗ Великі стартові витрати, ✗ DevOps навички потрібні, ✗ Обслуговування складне

🎯 AWS Bedrock vs vLLM: Коли що обирати?

Два найпопулярніших способи робити LLM inference в production. Вибір залежить від ваших потреб.

🟦 AWS Bedrock: Вибір для швидкості та Enterprise

Що це? Serverless API від AWS. Вам не потрібно думати про відеокарти, docker, драйвери.

Як виглядає код:

import boto3
client = boto3.client('bedrock-runtime')
response = client.invoke_model(
  modelId='anthropic.claude-3-5-sonnet-20241022-v2:0',
  body=json.dumps({'messages': [...]})
)

# Ви платите за токени. AWS масштабує сам.

Обирайте Bedrock, якщо:

  • ⏰ Time-to-Market понад усе — запустити продукт на вчора. Просто API call.
  • 📈 Масштабування — не ваша проблема — 100 користувачів сьогодні → 100K завтра, AWS сам виділить потужності
  • 🔗 Вже в екосистемі AWS — дані в S3, логіка в Lambda, Bedrock прямо інтегрується
  • 🔐 Потрібні закриті моделі — Claude 3.5 Sonnet, Llama 3.1 405B через API (не можна open-source)
  • 💼 Enterprise требует SLA — AWS гарантує uptime, compliance, monitoring

Вартість: ~$0.003 за 1K вхідних токенів (Claude 3.5 Sonnet)

⚡ vLLM: Вибір для контролю та економії

Що це? Двигун (engine), який ви ставите на свої GPU (або їх орендуєте)

Як виглядає код:

from openai import OpenAI
client = OpenAI(base_url="http://runpod-gpu:8000/v1", api_key="not-needed")
response = client.chat.completions.create(
  model="mistralai/Mistral-7B-v0.1",
  messages=[...],
  temperature=0.7
)

# Ви платите за орендовану GPU, не за токени

Обирайте vLLM, якщо:

  • 🔒 Максимальна приватність — дані не мають покидати ваш контур (Data Sovereignty, GDPR compliance)
  • 💰 Економіка на масштабі — мільйони запитів/день. vLLM на H100 (~$3K/місяць) дешевше, ніж платити за 1B токенів
  • 🎓 Fine-tuned моделі — ви навчили Llama розуміти юридичний сленг і хочете її обслуговувати
  • ⚙️ Контроль над всім — специфічні параметри PagedAttention, квантування, latency optimization
  • 💸 Бюджет — вторинний — вам не страшні 200K за сервер або DevOps

Вартість: ~$500/місяць (RunPod A100) до $5K+/місяць (Bare metal)

📊 Порівняльна таблиця (Decision Matrix)

Критерій AWS Bedrock 🟦 vLLM (Self-hosted) ⚡ Що обирати?

Економіка (1M запитів/день) ~$20K-50K/месяц (залежно від модель) ~$2K-5K/месяц (оренда GPU) Великий масштаб → vLLM

🚀 Чек-лист для вибору

Виберіть AWS Bedrock, якщо ви відповіли ДА на більшість:

  • ☑ Вам потрібна Claude 3.5 або інша closed-source модель?
  • ☑ У вас немає DevOps-инжнера на команді?
  • ☑ Сьогодні у вас <10K запитів на день?
  • ☑ Ви вже використовуєте AWS для інших сервісів?
  • ☑ Enterprise SLA важліше за ціну?

Виберіть vLLM, якщо ви відповіли ДА на більшість:

  • ☑ Ваші дані чутливі (банк, медицина, GDPR)?
  • ☑ У вас більше 100K запитів на день?
  • ☑ У вас є хороший DevOps/SRE?
  • ☑ Вам треба контроль над параметрами inference?
  • ☑ Ви планували fine-tune модель?

💡 Реальні приклади

Стартап Y-Combinator з 10 користувачами
→ AWS Bedrock. Часу запустити продукт немає, дані не чутливі.

Фінтех стартап (PSD2 compliant)
→ vLLM на bare metal. GDPR + дані в Європі + своя fine-tuned модель.

B2B SaaS з 50K активних користувачів
→ vLLM на RunPod з auto-scaling. Дешевше Bedrock при такому масштабі.

Корпоративний асистент у великій компанії
→ vLLM на bare metal в приватному датацентрі. Безпека > все.

🚀 Швидкий старт

Встановлення

pip install vllm

Запуск OpenAI-сумісного API сервера

vllm serve mistralai/Mistral-7B-Instruct-v0.1 \ --host 0.0.0.0 \ --port 8000 \ --tensor-parallel-size 2

Запуск з quantization (менше пам'яті)

vllm serve mistralai/Mistral-7B-v0.1 \ --quantization awq \ --gpu-memory-utilization 0.9

Тестування API

curl http://localhost:8000/v1/completions \ -H "Content-Type: application/json" \ -d '{ "model": "mistralai/Mistral-7B-v0.1", "prompt": "Що таке machine learning?", "max_tokens": 100 }'
